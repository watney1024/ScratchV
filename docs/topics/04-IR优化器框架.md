# Topic 04 — IR 优化器框架

> **难度**: 中级 | **源文件**: `scratchv/optimizer/` (5 passes)

---

## 是什么？

IR 优化器框架包含 5 个 IR 层面的优化 pass，它们共享相同接口：输入 `Program`，原地修改，返回变更计数。

```
优化前 IR → [ConstantFolder] → [DeadCodeEliminator] → [IRPeephole] → [MulAddFusion] → [LICM] → 优化后 IR
             常量折叠            死代码消除             窥孔优化        乘加融合         循环不变量外提
```

---

## 为什么？

代码生成器（指令选择）是**机械翻译**——它不管你给它什么 IR，都会忠实地翻译。优化器的作用是在翻译之前**先把 IR 变简洁**。

例如生成器可能产生：
```
%1 = const 2
%2 = const 3
%3 = add %1, %2     # 常量折叠 → %3 = const 5
%4 = mul %5, %1     # %1 还能继续折叠
%6 = add %7, 0      # 恒等消除 → mv %6, %7
```

如果不做优化，后端会为这些冗余指令生成真实的机器码。

---

## 核心概念

### 五个优化 Pass

#### 1. ConstantFolder（常量折叠）

编译时计算常量表达式：
```
%1 = load_const 2.0         →     %3 = load_const 6.0
%2 = load_const 3.0
%3 = add %1, %2
```

- 支持 ADD, SUB, MUL, DIV
- 要求全操作数为编译时常量
- 结果替换为 `LOAD_CONST`，后续指令可能继续被折叠

```python
class ConstantFolder:
    def _try_fold(self, instr):
        if instr.opcode not in (ADD, SUB, MUL, DIV):
            return None
        if not (lhs.is_constant and rhs.is_constant):
            return None
        result = self._compute(instr.opcode, lhs.value, rhs.value)
        return Instruction(LOAD_CONST, dest=instr.dest, attrs={"value": result})
```

#### 2. DeadCodeEliminator（死代码消除）

删除结果不被任何人使用的指令：
```
%1 = add %a, %b    →    (删除，因为 %1 未被使用)
%2 = mul %c, %d
store %2, %ptr
```

- 从输出/副作用指令回溯，标记所有"活的"指令
- 保留有副作用的指令（STORE, RETURN, BR, BR_IF, FOR, ENDFOR）
- 迭代执行直到不动点

#### 3. IRPeepholeOptimizer（IR 窥孔）

局部模式替换：

| 模式 | 替换 | 效果 |
|------|------|------|
| `add rd, rs, 0` | 删除 | 加 0 无意义 |
| `mul rd, rs, 1` | `mv rd, rs` | 乘 1 是恒等 |
| `mul rd, rs, 0` | `li rd, 0` | 乘 0 归零 |
| `j L` 后紧跟 `L:` | 删除跳转 | 冗余跳转 |

#### 4. MulAddFusion（乘加融合）

标记 `mul→add` 模式，为后端 FMA（Fused Multiply-Add）提供提示：
```
%1 = mul %a, %b           →    %2 = add %1, %c  ; fused_mul_add=True
%2 = add %1, %c
```
如果目标 ISA 支持 `fmadd`，后端可以发射单条指令。

#### 5. LICM（循环不变量外提）

将循环内不依赖循环变量的计算移到循环前：
```
for i in range(0, 100):        %pi = load_const 3.14
    %pi = load_const 3.14  →   for i in range(0, 100):
    %2 = mul %x, %pi               %2 = mul %x, %pi
endfor                        endfor
```

## 一步步

### Step 1: 运行单个优化 pass

```python
from scratchv.optimizer.constant_folding import ConstantFolder
from scratchv.frontend.dsl_parser import DSLParser

parser = DSLParser()
program = parser.parse_file("example.dsl")

print("Before optimization:")
# 打印 IR

folder = ConstantFolder()
changes = folder.optimize(program)
print(f"Folded {changes} constants")

print("After optimization:")
# 打印 IR
```

### Step 2: 链式运行多个 pass

```python
from scratchv.optimizer.constant_folding import ConstantFolder
from scratchv.optimizer.dead_code import DeadCodeEliminator
from scratchv.optimizer.licm import LICM

total_changes = 0
total_changes += ConstantFolder().optimize(program)
total_changes += DeadCodeEliminator().optimize(program)
total_changes += LICM().optimize(program)
print(f"Total optimizations: {total_changes}")
```

```python
from scratchv.compiler import PassManager

pm = PassManager(level="all")  # none | basic | all

```

### 死代码消除的回溯标记

```python
class DeadCodeEliminator:
    def optimize(self, program):
        used = set()
        # 从每条"有副作用的"指令开始回溯
        for instr in self._effectful_instructions(program):
            self._mark_used(instr, used)
        # 删除不在 used 集合中的指令
        for block in all_blocks:
            block.instructions = [i for i in block.instructions if i in used]
        return removed_count

    def _mark_used(self, instr, used):
        if instr in used:
            return
        used.add(instr)
        for op in instr.operands:
            self._mark_used(op.defining_instruction, used)  # 递归
```

### LICM 的循环不变量检测

一条指令是"循环不变量"当且仅当：
1. 它的所有操作数都不依赖于循环变量
2. 或者操作数是来自循环外的常量

```python
def _is_loop_invariant(self, instr, loop_var):
    for op in instr.operands:
        if op == loop_var:           # 直接依赖循环变量
            return False
        if not op.is_constant:       # 非常量，可能依赖循环变量
            # 检查 op 的定义指令是否也是不变量
            if not self._is_loop_invariant(op.defining_instruction, loop_var):
                return False
    return True
```

## 常见坑

| 坑 | 说明 |
|----|------|
| **Pass 执行顺序** | 顺序很重要。先折叠常量 → 再消除死代码（折叠后可能产生更多死代码）→ 再做 peephole。反过来可能遗漏优化机会 |
| **副作用指令** | STORE、BR 等不能被消除，即使它们的结果"看起来"没被使用 |
| **LICM 的保守性** | 不确定是否为不变量时保守处理（留在循环内），宁可少优化不可错优化 |
| **不动点迭代** | DeadCodeEliminator 和 ConstantFolder 都要迭代到不动点——删除一条指令可能让另一条变成死代码 |
