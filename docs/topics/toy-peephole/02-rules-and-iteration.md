# 02 规则详解与固定点迭代

## 回顾

上一篇介绍了窥孔优化的核心思想：用小窗口扫描汇编代码，把低效的指令组合替换成更精简的等价指令。这里我们来拆解 6 条规则的具体原理，理解滑动窗口如何工作，以及为什么一遍扫描不够。

## 六条规则详解

每条规则对应一种低效模式。理解它们为什么安全，比记住规则本身更重要。

### 规则 1: addi+addi 融合

两条连续的 `addi` 操作同一个寄存器链（第一条的结果是第二条的输入），直接合并，立即数相加。

```text
addi x1, x1, 3
addi x1, x1, 5
→ addi x1, x1, 8
```

安全原因：加法满足结合律，两次连续加法等价于一次加总。

### 规则 2: li+addi 融合

`li` 加载立即数后紧跟着 `addi` 做加法，合并为一条 `li`，在加载时直接算好最终值。

```text
li a0, 10
addi a0, a0, 5
→ li a0, 15
```

安全原因：常量折叠——编译器在编译期就能算出 `10 + 5 = 15`。

### 规则 3: beq x0 转 j

`beq x0, x0, label` 是条件跳转的一种特殊情况：x0 恒为零，比较结果永远为真。等价于无条件跳转 `j label`。

```text
beq x0, x0, end
→ j end
```

安全原因：编码更短（`j` 只需要目标地址，不用寄存器），语义完全一致。

### 规则 4: mv 交换消除

两条 `mv` 互为逆操作：`mv x, y` 把 y 复制到 x，紧接着 `mv y, x` 又把 x 复制回 y。恢复到原始状态，两条指令都是无用功。

```text
mv t0, t1
mv t1, t0
→ (两条都删除)
```

安全原因：不做任何改变。
注意：这只在两条 `mv` 连续时才成立，中间不能有其他指令修改寄存器。

### 规则 5: mv 链缩短

`mv a, b` 把 b 赋给 a，然后 `mv c, a` 又把 a 赋给 c。第二条直接改写为 `mv c, b`，跳过中间寄存器。

```text
mv t0, t1
mv t2, t0
→ mv t2, t1
```

安全原因：传递性——a 的值就是 b，所以 c 直接从 b 获取。

### 规则 6: addi 零消除

`addi rd, rs, 0` 加零不改变任何值。整条指令是冗余的。

```text
addi x2, x2, 0
→ (删除)
```

安全原因：加零恒等。

## 滑动窗口算法

窥孔优化器维护一个固定大小的窗口（通常 2-3 条指令），从位置 0 开始逐段扫描。

```
位置 0: [instr_0, instr_1] → 与规则 1-6 逐一比对
         ├─ 匹配成功 → 替换窗口内容，从位置 0 重扫
         └─ 都不匹配 → 窗口推进到位置 1
位置 1: [instr_1, instr_2] → 继续比对...
```

匹配成功后的**原地重扫**是关键：替换可能创造新的优化机会——比如合并两条 addi 后，新的 addi 和相邻的第三条 addi 可能再次匹配。

## 固定点迭代：为什么要跑多轮

原地重扫能处理"替换后同位置产生新机会"的情况。但如果替换产生的新机会出现在**更早的位置**呢？窗口已经过了那里，不会后退。

固定点迭代的解法很直接：把一整遍扫描当作一轮，重复执行，直到某轮完全没有改动为止。此时的代码不再有可优化的局部模式，称之为**固定点**。

### 三链 addi 的完整流程

```
第 1 轮扫描：
  [addi x1, x1, 3] [addi x1, x1, 5] [addi x1, x1, 2]
   ↑窗口匹配规则 1，合并前两条

  第 1 轮结果：
  [addi x1, x1, 8] [addi x1, x1, 2]
  新产生的 addi 和第三条相邻了——但窗口在位置 1 时已经检查过位置 0

第 2 轮扫描（从头开始）：
  [addi x1, x1, 8] [addi x1, x1, 2]
   ↑窗口再次匹配规则 1，合并

  第 2 轮结果（固定点）：
  [addi x1, x1, 10]
  三条指令变一条，节省了 2 条。
```

第 1 轮合并完以后，窗口已经过了位置 0，不会自动回去。第 2 轮从头开始，能完整覆盖上一轮留下的新机会。

### 为什么设定 50 轮上限

理论上固定点应该在几轮内达到。但为了防止两条规则互相触发形成无限循环（比如规则 A 把指令 X 变成 Y，规则 B 又把 Y 变回 X），必须设一个硬性上限。50 轮是一个既保证安全、又不会掩盖真正 bug 的经验值。

## 如何读优化报告

执行 `demo03_full --report` 会输出一份报告，包含四个信息块：

```
Instructions before: 7        ← 优化前的指令数
Instructions after:  3        ← 优化后的指令数
Saved:              4 (57%)   ← 节省量和百分比

Rules applied:                ← 每条规则的命中统计
  addi+addi fusion: 1
  beq zero-zero to j: 1
  mv swap elimination: 1
  addi zero elimination: 1

Diff (changed lines only):    ← 逐行变更对比
  + 代表新增，- 代表删除或修改
```

阅读时关注三层：

- **总数对比**：`before → after` 的变化幅度。57% 的节省说明原始代码包含大量冗余。
- **规则分布**：哪些规则贡献最大。如果某条规则命中数为 0，可以思考——是不是代码生成器本来就没有产生那种模式？
- **逐行变更**：验证每处修改都是正确的。如果一条预期命中的规则没有出现在 diff 里，可能是条件约束没满足。

## 下一步

6 条规则已经覆盖了最常见的汇编优化模式，但窥孔优化的设计空间远不止于此。扩展练习将引导你添加新规则（比如 `mul x, 2^n → slli x, n`）、实现指令成本模型，以及将优化器集成到真正的编译管线中。
