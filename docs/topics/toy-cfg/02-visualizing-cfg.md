# 02 可视化 CFG：看图理解控制流

## 为什么需要可视化？

基本块和边的概念理解了，但光看 IR 文本很难一眼看出程序的整体结构。一张图胜过千言。CFG 可视化把抽象的控制流变成直观的方框和箭头，让你快速回答这些问题：

- 程序有哪些执行路径？
- 哪些块是分支的岔路口？
- 哪些块是路径的交汇点？
- 有没有循环？（箭头往回指了）

## DOT 语言极简入门

ScratchV 使用 Graphviz DOT 格式描述 CFG 图。DOT 是一种纯文本的图描述语言，核心就三样东西：

| 元素 | DOT 写法 | 说明 |
|------|----------|------|
| **节点** | `"entry" [shape=box]` | 一个方框，代表一个基本块 |
| **边** | `"entry" -> "then"` | 一个箭头，代表一条控制流 |
| **属性** | `[color=blue, style=dashed]` | 方括号里写样式，修饰节点或边 |

一个完整的 DOT 文件就长这样：

```
digraph "CFG_demo" {
  rankdir=TB;
  "entry" [label="entry", fillcolor=lightgreen, style=filled];
  "entry" -> "then" [color=blue, style=dashed];
}
```

把它喂给 `dot` 命令（`dot -Tpng cfg.dot -o cfg.png`），就生成一张 PNG 图片。

## 颜色和图例

我们的 CFG 可视化用了一套统一的颜色规则，看懂这套规则，任何 CFG 图你都能秒读：

```
┌──────────────────────────────────────────────────────┐
│                    图例（Legend）                      │
│                                                      │
│   ┌───────┐   入口块（lightgreen 背景）               │
│   │ entry │   每个函数唯一，图的最顶端                  │
│   └───┬───┘                                           │
│       │                                               │
│   ┌───▼───┐   出口块（lightcoral 背景）               │
│   │  end  │   函数返回处，0 个出口                     │
│   └───────┘                                           │
│                                                      │
│   普通块（白色背景）                                    │
│                                                      │
│   ─ ─ ─ ▶   条件分支（蓝色虚线）                      │
│             标签标注条件（true / false / body）        │
│                                                      │
│   ──────▶   无条件跳转（红色实线）                     │
│             一定会走这条路                             │
└──────────────────────────────────────────────────────┘
```

**记住三点就够了**：绿色是入口，珊瑚色是出口，蓝色虚线是"二选一"，红色实线是"一直走"。

## 三个典型场景

### 场景 1：最简单的程序（entry → exit）

一个没有分支的程序，比如 `return 42`。CFG 只有两个块，一条边：

```
┌──────────┐
│  entry   │  绿色 = 起点
│  ret 42  │
└────┬─────┘
     │  红色实线（无条件跳转）
┌────▼─────┐
│   end    │  珊瑚色 = 终点
│  ret 42  │  （0 个出口）
└──────────┘
```

这种"直线型"CFG 没什么好分析的，就是顺序执行到底。

### 场景 2：if-else 分支

条件分支产生了岔路。entry 块根据条件选择走 then 还是 else，然后汇合到 end：

```
              ┌──────────┐
              │  entry   │
              │ br_if $c │
              └┬──┬──┬───┘
      true    │  │  │  false
     ┌────────┘  │  └────────┐
     ▼           │           ▼
  ┌──────┐ 蓝色虚线│      ┌──────┐
  │ then │◄───────┘      │ else │
  │ add  │  条件分支      │ sub  │
  │ br   │               │ br   │
  └──┬───┘               └──┬───┘
     │  红色实线             │  红色实线
     │  无条件跳转            │  无条件跳转
     └──────┬───────┬───────┘
            ▼       ▼
         ┌──────────┐
         │   end    │
         │   ret    │
         └──────────┘
```

解读要点：entry 有两个出口，说明这里是决策点。then 和 else 各有两条边（一出一进），说明它们分叉后又汇合了。如果出口块有多个前驱，那它就是"汇聚点"。

### 场景 3：while 循环

循环的标志性特征：**箭头往回指**。

```
               ┌──────────┐
               │  entry   │
               │ br_if $c │
               └┬──┬──┬───┘
       body    │  │  │  exit
      ┌────────┘  │  └────────┐
      ▼           │           ▼
  ┌──────┐  蓝色虚线│      ┌──────┐
  │ body │◄────────┘      │ end  │
  │ ...  │  条件："body"   │ ret  │
  │ br   │               └──────┘
  └──┬───┘
     │  红色实线
     │  跳回 entry（循环！）
     └──────▶
```

看到 `body → entry` 这条**逆向的红色实线**，就知道这里有循环。entry 既是循环入口也是循环判断，body 执行完跳回去重新检查条件。这是"先判断再执行"的结构。

如果看到多条箭头指向同一个上层块，那可能是嵌套循环或多层回边。

## 怎么读 CFG 图？

拿到一张 CFG 图，按这个顺序看：

1. **找绿色块**：那是入口，读图从这里开始
2. **数出口**：找所有珊瑚色块，看程序在哪里返回
3. **找岔路**：蓝色虚线边就是分支，看标签知道什么条件下走哪条路
4. **找回路**：红色实线如果指向**上方**的块，说明有循环
5. **顺路径**：把入口到出口的可能路径走一遍，确认理解正确

读完一张图，程序的整体结构就在你脑子里了。
