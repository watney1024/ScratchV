# 00 什么是控制流图（CFG）？

## 从线性代码到图

你写的程序是一行一行按顺序排列的指令。但计算机执行的时候，它不会永远傻傻往下走——遇到条件判断（if/else）它会二选一，遇到循环（for/while）它会回头重复跑。

如果只用"第几行到第几行"来描述这种跳跃，很难看清程序的全貌。控制流图（Control Flow Graph, CFG）就是解决这个问题的：它把线性的指令流变成一张**有向图**。

```
原始代码（线性）                   控制流图（图结构）
  if x > 0:                        ┌──────────┐
    y = 1                ┌──────→│  入口    │
  else:                  │       │  if x>0  │──┐
    y = -1               │       └────┬─────┘  │
  return y               │     真    │   假    │
                         │    ┌──────▼──┐ ┌───▼──────┐
                         │    │  真分支  │ │  假分支  │
                         │    │  y=1    │ │  y=-1    │
                         │    └──┬──────┘ └──┬───────┘
                         │       │           │
                         │       └─────┬─────┘
                         │          ┌──▼──────┐
                         └─────────→│  出口   │
                                    │ return y│
                                    └─────────┘
```

图中的每个**节点**叫"基本块"——一段直线执行的指令，进去之后必须走到底，中途不拐弯。每条**边**代表一次跳转：可能是条件分支、无条件跳转，或者自然顺序执行（FALLTHROUGH）。

## 为什么需要 CFG？

有了图，编译器就能"看见"代码的结构：

- **发现不可达代码**：从入口出发走不到的块，可以直接扔掉。比如 return 之后还跟着的指令。
- **看清循环**：图中出现"回边"（从后块跳回前块）的地方就是循环。编译器知道哪里在反复执行，才能做循环展开、软件流水等优化。
- **指导优化顺序**：很多优化（常量传播、死代码消除）需要知道"从 A 能不能走到 B"。CFG 让这种查询变成一次图上搜索，而不是从头扫描指令。

没有 CFG 的编译器就像没有地图的导航——指令排成一排，但你看不出哪条路通哪条路。

## 控制流在 IR 里怎么表示？

ScratchV 的中间表示（IR）用几条指令表达控制流：

- **BR**：无条件跳转。走到这里就跳到另一个块。
- **BR_IF**：条件分支。根据条件真/假，走两条路之一。
- **FOR / ENDFOR**：循环的开始和结束。FOR 有两个出口（进入循环体 或 跳过循环），ENDFOR 有一个出口（跳回 FOR 头）。
- **RETURN**：函数返回，没有出口。

每条这样的指令都对应 CFG 中的一条或多条边。解析 IR 的一个核心任务，就是找到这些"边界"，把指令切成块。

## 常见误区

### FALLTHROUGH 不是跳转

如果块末尾没有 BR/BR_IF 这类跳转指令，执行会自然流到下一个块。这叫 FALLTHROUGH。它**不是一条指令**，只是"刚好排在后面"。新手最容易犯的错误：把 FALLTHROUGH 当成 JUMP 来处理。两者本质不同——FALLTHROUGH 不需要花指令，JUMP 要。

### 连续标签之间可能有空块

```
.La:
.Lb:
  add x1, x2, x3
```

标签 `La` 和 `Lb` 之间没有指令，会形成一个空的基本块。正常编译器前端不会制造这种冗余，但分析器必须能正确处理。

### 分支之后的指令不会执行

```
  ret v1
  add v2, v1, v3   # 这条永远不会跑
```

return 之后的指令虽然还躺在 IR 里，但它所在的块已经不可达。CFG 帮你一眼看到这个问题。
