Topic 18 — 指令调度器（基本块内列表调度）

难度: 入门 | 源文件: scratchv/backend/inst_scheduler.py | 行数: ~552

是什么？

指令调度器（inst_scheduler.py）重新排列一个基本块内的指令顺序，以减少 CPU 流水线停顿（pipeline stall）。

核心思想：如果指令 B 依赖指令 A 的结果，而 A 需要 3 个周期才能算完，那么只要不破坏依赖关系，可以把不相关的指令 C/D/E 插入到 A 和 B 之间。

调度前（有空泡）:           调度后（填满空泡）:
  mul t0, a0, a1  (3cycle)    mul t0, a0, a1  (3cycle)
  add t1, t0, a2  ← 等 2 拍    lw  t2, 0(sp)   ← 不依赖 t0
  lw  t2, 0(sp)                addi sp, sp, -16 ← 也不依赖
  addi sp, sp, -16              add t1, t0, a2  ← t0 已经好了

为什么？

现代 CPU 使用流水线技术并行处理多条指令。但数据依赖会导致"气泡"——后续指令必须等前面指令的结果。

时间 →
取指: [mul] [add] [气泡] [气泡] [lw] ...
译码:       [mul] [add] [气泡] [气泡] ...
执行:             [mul---] [add] [气泡] ...
写回:                      [mul---] [add] ...

指令调度的目标就是把气泡填满，让每个时钟周期都有指令在执行。

核心概念

1. 指令延迟模型

不同指令的执行周期不同：

指令类型 延迟（周期） 例子
简单 ALU 1 add, sub, xor, and
乘法 3 mul, mulh
除法 16 div, rem
内存 Load 2 lw, lh, lb
内存 Store 0 sw, sh, sb（非阻塞）
分支 1 beq, bne
跳转 0 j, jal, ret

2. 数据依赖 DAG

调度器首先构建一个有向无环图（DAG），节点是指令，边表示依赖：

A: mul t0, a0, a1     ┌───┐
B: add t1, t0, a2     │ A │ (写 t0, 延迟 3)
C: lw t2, 0(sp)       └─┬─┘
D: addi sp, sp, -16      │ (B 读 t0)
                       ┌─▼─┐
                       │ B │
                       └───┘
                       ┌───┐   ┌───┐
                       │ C │   │ D │  (独立，可调度到 A 和 B 之间)
                       └───┘   └───┘

3. 列表调度算法

1. 构建依赖 DAG，计算每个节点的优先级（关键路径长度）
2. 重复直到所有节点都调度完毕：
   a. 找出所有"就绪"节点（所有前置依赖已调度完）
   b. 按优先级排序（关键路径长的优先）
   c. 调度优先级最高的节点
   d. 更新时钟

一步步

Step 1: 基本使用

from scratchv.backend.inst_scheduler import (
    InstructionScheduler, parse_instructions
)

asm_text = """
    mul t0, a0, a1
    add t1, t0, a2
    lw  t2, 0(sp)
    addi sp, sp, -16
"""

instructions = parse_instructions(asm_text)
sched = InstructionScheduler()
dag = sched.build_dag(instructions)
scheduled = sched.schedule(dag)

for inst in scheduled:
    print(f"  {inst.opcode} {', '.join(inst.operands)}")

# 输出调度报告
print(sched.report(instructions, scheduled))

Step 2: 命令行使用

python -m scratchv.backend.inst_scheduler input.s -o scheduled.s --report

Step 3: 自定义延迟模型

my_latency = {
    "add": 1, "sub": 1,
    "mul": 5,     # 假设你的 CPU 乘法是 5 周期
    "lw": 3,      # 内存延迟更高
    # ...
}
sched = InstructionScheduler(latency_model=my_latency)

代码走读

DAG 构建（核心）

def build_dag(self, instructions):
    last_def = {}  # 每个寄存器最后一次被谁定义
    for inst in instructions:
        # RAW 依赖：每条指令的 use 依赖于最近的 def
        for use_reg in inst.uses:
            if use_reg in last_def:
                pred = last_def[use_reg]
                latency = self._get_latency(pred.opcode)
                node.predecessors.append((pred, latency))
        # 更新 def 表
        for def_reg in inst.defines:
            last_def[def_reg] = node

优先级计算（关键路径）

def _compute_priorities(self):
    # 从叶子节点向根节点反向遍历
    for node in reversed(topo_order):
        max_succ_prio = 0
        for succ, lat in node.successors:
            max_succ_prio = max(max_succ_prio, succ.priority + lat)
        node.priority = max_succ_prio + self._get_latency(node.opcode)

优先级 = 从该节点到任意叶节点的最长路径（按延迟加权）。优先级越高的指令越要先调度——因为它在关键路径上。

列表调度主循环

while remaining:
    # 收集就绪节点
    ready = [n for n in remaining if all(p.scheduled for p in n.predecessors)]
    # 按优先级降序排序
    ready.sort(key=lambda n: (-n.priority, n.inst.id))
    # 选最优的调度
    node = ready[0]
    node.scheduled = True
    result.append(node.inst)

动手练习

练习 1: 手写一个需要调度的例子

写 5-6 条有数据依赖的指令，手动画出依赖 DAG，然后手动调度。最后用代码验证你的调度结果。

练习 2: 比较调度前后的周期数

用 sched.report() 查看调度前后的估算周期差异。对于简单的基本块，调度能有多大提升？

练习 3: 调整延迟模型

把乘法延迟从 3 改为 10，观察调度结果的变化。延迟越长，调度空间越大还是越小？

常见坑

坑 说明
只调度基本块内 跨基本块的调度（全局调度）需要更复杂的分析，当前只做块内调度
延迟模型简化 实际 CPU 的流水线比这个模型复杂得多（乱序执行、超标量、cache miss 等），这里的估计只反映趋势
Store 延迟为 0 假设 Store 是非阻塞的（CPU 有 store buffer），后续 load 可以不等它完成
寄存器重名 同一寄存器在不同生命周期可能代表不同值，简单 DAG 可能产生假依赖（WAW/WAR）
控制流指令 分支和跳转通常放在块末尾，不应被调度到块中间
